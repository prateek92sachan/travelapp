01. Google Places search radius clamp

Bug fixShipped
Trigger
User searched "Kodagu" — no pins rendered. Network tab showed places:searchText returning 400 INVALID_ARGUMENT with message "Invalid circle.radius. Radius must be in the ra…" (truncated).
Root cause
Google Places searchText API caps locationBias.circle.radius at 50,000 m. The destination radius was being computed from the geocode bounding box and clamped at 300 km (src/hooks/useTrip.jsx:387) — too large for the API.
Fix
Clamp at the API call site so all callers of placesTextSearch are covered:
radius: Math.min(50000, Math.max(1, radiusMeters))
at src/services/googleMaps.js:259.
Files
src/services/googleMaps.js

Usability

Restored. Any destination whose geocode diagonal exceeds 100 km (states, regions, large districts) now returns results instead of failing silently with no pins.

Performance

Neutral. Same one network call per search; just no longer wasted on a 400.

Sequence

Unchanged. Internal radius (used for tab queries + distance filters) still tolerates 300 km — only the API request body is clamped.

02. AI place summary v1 — Claude (Wiki + web search)

DeprecatedFunctions removed
Trigger
User asked why the detail card for "Big Cup Cafe" did not show an AI summary. By design at the time, summaries required a Wikipedia extract (cafes have none).

Two Cloud Functions:

  • webSearchPlace — Claude Haiku 4.5 + web_search_20250305 tool, max 2 searches, summarizes places without Wikipedia.
Hook useClaudeSummary picked Wiki path if extract present, web-search path otherwise. Forever positive cache + 24 h negative cache per placeId. Source pill (AI summary · Web / · Wikipedia) shown in card head.

Resolution
Functions later removed (see T1); hook replaced by usePlaceSummary (see change 03).

Usability

Zero net effect — the feature was online for a few minutes, never returned a successful summary, then removed.

Performance

Cloud Functions had cold-start cost paid until deletion. Now zero (deleted in T1).

Sequence

Removed entirely. No card path calls Anthropic any more.

03. AI place summary v2 — Gemini 2.0 Flash + Google Search grounding

Live
Trigger
Replacement for change 02 after Anthropic credits ran out. Free tier required.
Choice
Gemini 2.0 Flash via @google/genai SDK with tools: [{ googleSearch: {} }]. Free tier: 1500 requests/day, 15 RPM. Separate key from the existing Google Cloud key (issued at aistudio.google.com/apikey).
Backend
New Cloud Function geminiSearchPlace in functions/index.js. System prompt enforces 40–50 word output, sensory tone, no preamble, and a literal NO_INFO token when grounding turns up nothing usable. Auth required. Per-user daily cap added in T1.
Frontend hook
New file src/hooks/usePlaceSummary.js (replaces useClaudeSummary.js, deleted). Signature (placeId, name, hasWiki). Bot fires only when no Wikipedia article — Wiki-backed places show the Wiki extract directly via the existing description block.
Cache
Positive: travelapp:place-summary:{placeId}, forever. Negative: travelapp:place-summary:neg:{placeId}, 24 h. Retry button clears the negative cache.
UI
Pill AI summary · Web appears in detail-card head when ready. Skeleton during loading. Retry on error.
Files
functions/index.js, functions/package.json, src/hooks/usePlaceSummary.js (new), src/components/TabbedPlacesWidget.jsx, src/styles/global.css

Usability

Wiki-less places (cafes, shops, niche POIs) now get a 40–50 word grounded summary in the detail card. Wiki-backed landmarks unchanged (extract shown directly, no AI block). Failure modes surface as a retry button rather than a silent empty card.

Performance

One Cloud Function invocation per unique placeId for the lifetime of the cache. Negative cache eliminates repeat billing on broken places. Free-tier quota covers a single dev easily; the per-user 50/day cap added in T1 bounds abuse.

Sequence

Detail card opens → fetchWikiSummary debounced 300 ms → if wikiExtract ≥ 30 chars then render extract; else hook calls geminiSearchPlace → Gemini issues 1–3 web searches → returns summary string (or NO_INFO) → write through to positive cache or negative cache → UI repaints.

Bug fixShipped
Trigger
React warning Encountered two children with the same key, "ChIJR3122B9u5kcRaCck3PlB9DM" on the world-map pin grid.
Root cause
The pin loop used key={pt.placeId}. allPoints was deduped per list, but a place that exists in two wishlists produced two entries — same placeId, different listId. React saw a collision.

05. Audit pass T1 — Backend hardening + Gemini rate limit

Shipped
Scope
Cloud Functions cleanup, error sanitization, runtime hardening, abuse protection.
Changes
  1. Deleted orphan functions summarizePlace + webSearchPlace (source and deployed Cloud).
  2. Dropped @anthropic-ai/sdk + axios dependencies; removed ANTHROPIC_API_KEY secret binding.
  3. Sanitized error messages — internal err.message no longer echoed to the client. Returns generic 'Summary service temporarily unavailable.' / 'Billing data temporarily unavailable.'.
  4. Added globalOptions overrides: memory: '256MiB', concurrency: 40, timeoutSeconds: 30 floor. getCostBreakdown floors to 20 s (BQ should never block longer).
  5. Per-user daily Gemini rate cap: 50/day per uid, enforced via a Firestore atomic transaction at users/{uid}/usage/{YYYY-MM-DD} (gemini counter field). Exceeding cap returns resource-exhausted.
  6. Input length cap on name and destination — 200 chars each — prevents prompt-stuffing abuse.
  7. BigQuery query — added _PARTITIONTIME >= TIMESTAMP_TRUNC(CURRENT_TIMESTAMP(), MONTH) to skip historical partitions, and 5-min in-memory result cache so a refreshing dashboard doesn't scan the export table on every load.
Files
functions/index.js, functions/package.json

Usability

Dashboard load: faster (cache hit) most of the time. AI summary user sees nothing different until they exceed 50/day, at which point an error surfaces a retry button (cap resets at UTC midnight per the day-key shape).

Performance

BigQuery scan size and bytes-billed drop materially — partition prune + 5-min cache means dashboard re-renders cost zero BQ. Cold-start improves (fewer deps to load).

Sequence

Gemini call now traverses: auth check → input validate + truncate → Firestore txn (read+write counter, throw on overflow) → Gemini API → return. The txn adds ~30–80 ms server-side; trivial vs. the ~1.5–4 s Gemini call.

06. Audit pass T2 — Wikipedia service dedup + concurrency cap

Shipped
Scope
src/services/wikipedia.js
Changes
  1. In-flight dedup. Module-level INFLIGHT = new Map() keyed by name + context. fetchWikiSummary returns the existing promise when one is pending — collapses parallel duplicates fired by enrichWithWiki + backfillPhotosWithWiki hitting the same query.
  2. Concurrency cap. Added mapWithConcurrency(items, 5, fn) — bounded worker pool. Promise.all over up to 50 viewport pins (potentially 100 Wiki API calls) replaced with 5 workers draining a queue.

Usability

Subjective: slightly snappier on dense viewports because the browser stops thrashing on 50+ parallel fetch calls. Final photos still land, just in a more orderly fashion.

Performance

Up to 2× fewer Wikipedia API calls in the typical case (overlapping tab + viewport enrichment paths converged). Network stack stays under the browser's per-host concurrent connection limit so Wikipedia calls don't queue behind other traffic.

Sequence

Same overall pipeline. Internally: request → INFLIGHT check → cache check → fetch (search + summary) → write cache → resolve all awaiters → delete from INFLIGHT. Bulk enrichers (enrichWithWiki, backfillPhotosWithWiki) now wait for free worker slots rather than firing all at once.

07. Audit pass T3 — Query gating + cache-key limit

Shipped
Scope
src/hooks/queries/useTabQuery.js, src/services/googleMaps.js, src/services/tmapService.js
Changes
  1. Stale-enrichment gate. useTabQuery.js background enrichWithWiki previously wrote results into queryClient.setQueryData(key, …) unconditionally. Now: capture the initial stashed array reference at fetch time; on enrichment resolve, write back only if queryClient.getQueryData(key) === stashed still holds. Prevents overwriting fresher data from a subsequent refetch / destination change.
  2. Cache-key includes limit. Google + Tmap viewport caches keyed by {lat, lng, radiusMeters, category, bounds}; now also includes the requested limit. Previously: a request for limit: 10 served back a cached limit: 5 array (length mismatch silently shipped 5 items).

Usability

Eliminates a rare-but-real bug class — fast destination switching could have replaced a fresh result with stale Wiki enrichment from the old destination. Users won't notice in steady state; they would have noticed eventually as "why is this place showing the old city's photo".

Performance

Cache-key fix prevents under-fetching (cached 5-item array returned where 10 was asked) — surfaces with one extra Places call when a tab demands more items than the seed query did, but eliminates incorrect short results.

Sequence

Initial tab fetch → return stashed (photo URLs hidden) → background enrichWithWiki → identity check against cache (new) → write enriched data only if cache still holds the same reference.

10. Audit pass T7 — useTrip re-render mitigation

ShippedLargest change
Problem
TripContext value was an object with ~40 keys + a 30-dependency useMemo. Any state change rebuilt the value identity, which forced every useTrip() consumer (Header, EmptyStateGlobe, PlacesDrawer, map providers, App's root view, etc.) to re-render — even when none of the fields they actually read had changed.
Approach chosen
"Lighter path" — leave TripContext in place; introduce a second, minimal context just for the search orchestrator function; migrate every consumer to read everything else directly from the zustand stores. End state: nothing subscribes to TripContext anymore.
Implementation
  1. New context. TripSearchContext = createContext(null) + useTripSearch() hook. Provider wraps it around TripContext.Provider. Value is useMemo(() => ({ search }), [search]) — and search's useCallback deps are all stable zustand setters, so the value identity is stable across normal renders.
  2. Consumer migration — every useTrip() call replaced with direct selectors + useTripSearch():
    • src/App.jsx — { coords, error } → useSearchStore; { search } → useTripSearch.
    • src/components/Header.jsx — { destination, setDestination, date, setDate, loading, coords } → useSearchStore; { search } → useTripSearch.
    • src/components/EmptyStateGlobe.jsx — { search } → useTripSearch.
    • src/components/PlacesDrawer.jsx — { activeTab, switchTab } → useSearchStore.
    • src/components/map/useMapData.js — { selectPlace, clearViewportItems, search } → direct store reads + useTripSearch.
  3. Store action lift. clearViewportItems (formerly a wrapped useCallback inside TripProvider that fired a window event after clearing state) moved into mapStore as a first-class action so consumers can pull it directly.
Files
src/hooks/useTrip.jsx, src/stores/mapStore.js, src/App.jsx, src/components/Header.jsx, src/components/EmptyStateGlobe.jsx, src/components/PlacesDrawer.jsx, src/components/map/useMapData.js

Usability

No visual change, but interactions feel marginally crisper — typing in the search input, switching tabs, toggling map layers no longer force the entire app tree through a re-render. Most noticeable on the Header search input and the tab drawer on mobile.

Performance

Each consumer now subscribes only to its own zustand slice (or to the small { search } context). zustand selectors compare by Object.is so a render only fires when that field actually changes. Component re-render count per typical interaction (search input keystroke, map pan, tab switch) drops materially — order-of-magnitude reduction for any tree whose root was the bloated context consumer.

Sequence

Before: store mutation → TripProvider re-renders → useMemo recomputes context value → all 5 useTrip() consumers re-render → their children re-render. After: store mutation → only the zustand subscribers whose selected slice changed re-render → minimal subtree updates. The orchestrator search() function is still invoked the same way; only the subscription topology changed.
